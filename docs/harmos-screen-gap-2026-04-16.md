# HARMOS勤怠 画面ベース機能差分メモ

作成日: 2026-04-16

## 前提

- 2026-04-16に、管理者アカウントでHARMOS勤怠の実画面を確認しました。打刻、登録、申請、承認、差し戻し、月締、CSV/PDF出力など、データが変わる操作は実行していません。
- このアプリ側は `src/backend-laravel-app`, `src/admin-web/next-app`, `src/windows-punch/StaffHub.PunchApp` の実装と既存設計書を見て確認しました。
- ユーザー指定により、`シフト` は今回の不足調査対象から外しています。
- 当園向けに不要と決めている `フレックスタイム`, `複雑な勤務体系`, `複数拠点向けの汎用設定` は、HARMOSにあっても必須不足には含めず、別枠の「必要なら後回し」にしています。

## HARMOS側で画面確認できる主な領域

公式ヘルプ上では、勤怠メニューとして `打刻`, `日次勤怠`, `日次勤怠修正`, `シフト`, `勤怠承認`, `レポート`, `勤怠の予実管理` が並び、システム管理メニューとして `お知らせ`, `打刻履歴`, `勤務区分`, `休暇設定`, `休暇管理`, `日次勤怠・項目設定`, `勤怠設定` が確認できます。さらに有料機能として `有給休暇等の自動付与`, `時間有給休暇`, `子の看護・介護休暇`, `届出`, `勤怠アラート`, `オリジナル時間帯集計` もあります。

給与明細側は `給与、賞与、源泉徴収票`, `データ定義`, `データCSV登録` が主な画面です。

## 画面別の細かい差分

### 1. 打刻

HARMOSはブラウザ打刻、ICカード打刻、LINE/Slack/LINE WORKS、QRコード、位置情報、夜勤打刻まで持っています。このアプリは `Windows + RC-S380 + ICカード` に絞っており、当園運用としてはむしろ安定しています。

不足として見るなら、休憩開始/終了の打刻、打刻位置、夜勤打刻、スマホ打刻です。ただし、当園で出退勤だけをカードで取るなら優先度は高くありません。

### 2. 日次勤怠/日次勤怠修正

一番大きい不足はここです。HARMOSは「実打刻を残しながら、日次勤怠修正画面で勤務区分・出勤・退勤・休憩・備考を直す」運用が画面として明確です。このアプリは日次一覧と承認は強くなっていますが、日々の修正フォームはまだHARMOSほどはっきりしていません。

移行時に現場で困るのは、打刻忘れ、打刻時刻の補正、休憩修正、承認後の差戻し後再修正です。ここは優先度Aです。

### 3. 勤怠承認/月締め

HARMOSは日次承認、差戻し、承認取消、月締申請、月締確定、月締解除が一連の流れです。このアプリは承認/差戻し、一括処理、月締め/解除があるのでかなり近いです。

不足は `承認取消` と `職員本人の月締申請` です。園の運用を「管理者が月末に確認して締める」にするなら、月締申請は省略してもよいです。

### 4. 届出/休暇

HARMOSは届出画面に申請タブ/承認タブがあり、承認後に日次勤怠の勤務区分へ反映されます。このアプリは休暇申請、承認/差戻し/却下、取消、有給残数、台帳まであり、当園向けにはHARMOSより説明しやすい設計です。

不足は、届出区分ごとの入力項目制御です。たとえば直行直帰なら出勤/退勤時刻、残業申請なら残業上限時刻、有給なら半日区分、という出し分けはまだ弱いです。

### 5. レポート

HARMOSは今日の出退勤、勤怠エラー、勤怠アラート、月締状況、月次、残業管理、36協定管理までレポートが広いです。このアプリは今日の出退勤、承認履歴、有給、日次/月次CSV/PDFがあり、移行に必要な最低限は揃っています。

足すなら `勤怠エラーレポート` と `月締状況レポート` が最優先です。打刻漏れだけでなく、休憩不足、未承認、未申請、未退勤、長時間勤務、月締前残件を一覧で出せると、HARMOSからの移行感がかなり近づきます。

### 6. 給与明細

給与/賞与はかなり近いです。むしろこのアプリはCSVからPDFを自動生成できるため、当園運用ではHARMOSより楽になる可能性があります。

不足は `源泉徴収票` と `詳細なデータ定義`、`未閲覧確認`、`公開前プレビュー` です。給与・賞与の毎月運用だけならすぐ必須ではありませんが、年末までには源泉徴収票を検討した方がよいです。

### 7. システム管理

部門、拠点、雇用形態、勤務区分、申請区分、休暇区分、有給付与、勤怠アラート、日次項目は入っています。HARMOSほど汎用ではありませんが、当園専用としては十分に寄っています。

不足は `設定が実処理にどこまで効いているか` の確認です。実画面では、日次項目の表示/非表示、CSV出力有無、勤務区分ごとの法定集計ルール、36協定、通知メール、オリジナルCSVまで細かく設定できます。マスタ画面を作るだけでなく、日次集計・月次集計・CSV/PDF・承認画面へ反映されることが重要です。

   - 締め済み月は編集不可

2. 勤怠エラーレポート
   - 打刻漏れ、未退勤、休憩不足、長時間勤務、未承認、未申請
   - 対応済み/未対応の状態

3. 月締状況レポート
   - 社員別の未申請数、承認待ち数、承認済み数、月締状況
   - 月締前の残件確認、CSV/PDF出力導線

4. 承認取消
   - 日次承認済みを承認待ちへ戻す
   - 取消理由と監査ログ

5. 届出モデルの拡張
   - 勤務区分等変更、時間休暇、残業上限時刻、期間指定、承認者/承認履歴
   - 日次勤怠への反映と申請取消

6. 勤務区分・日次項目設定の実処理連動
   - 勤務区分ごとの休憩/所定/深夜/法定集計ルール
   - 権限別表示、CSV出力有無

7. 給与明細の運用強化
   - 公開前プレビュー
   - 未閲覧一覧
   - 源泉徴収票種別の追加

8. 月締め運用の最終決定
   - HARMOS互換: 職員月締申請 + 管理者確定
   - 当園簡素版: 管理者が全員分を確認して月締め

## 改善案

HARMOSの全機能を完全コピーするより、当園の現場で迷いが出る部分を先に埋める方がよいです。特に `打刻後の日次修正`, `承認前後の見え方`, `月締前の残件確認`, `休暇/届出の反映` を優先すると、日々の運用不安がかなり減ります。

### 改善方針

| 方針 | 内容 |
| --- | --- |
| HARMOS互換に寄せるところ | 日次勤怠、承認、届出、月締状況、勤怠エラー、休暇残数 |
| 園専用に簡素化するところ | シフト、複数拠点、フレックス、複雑な勤務形態、LINE/Slack/QR打刻 |
| 強く残すところ | Windows ICカード打刻、カード割当、端末運用、給与/賞与PDF生成 |
| 設定より固定運用でよいところ | 36協定の詳細設定、多段承認、オリジナル時間帯集計は必要になってから追加 |

### フェーズ1: すぐ効く運用改善

| 改善 | 目的 | 画面案 | データ/処理案 |
| --- | --- | --- | --- |
| 日次勤怠修正フォーム | 打刻忘れ・時刻補正・休憩修正を迷わず処理する | 管理画面の日次行に `修正`、詳細に `実打刻` と `修正後` を並べる | `attendance_daily_adjustments` または日次テーブルに修正値、修正理由、修正者、修正日時を保持 |
| 休憩明細 | 合算休憩だけでは説明できないケースを扱う | 休憩1/休憩2、追加ボタン、合計休憩を自動表示 | `attendance_break_segments` を追加。最低限は開始/終了/翌日/手入力フラグ |
| 日次履歴タブ | 誰が何を変えたかを説明できるようにする | 日次詳細に `履歴` タブ | 既存監査ログを日次IDで引けるようにし、項目単位の差分を表示 |
| 承認取消 | 承認済みの誤りを戻せるようにする | 承認済み行に `承認取消` | 締め済み月は禁止。取消理由を監査ログへ保存 |
| 勤怠エラーレポート | 月末に問題を探す手間を減らす | `勤怠エラー` 画面を追加 | 打刻漏れ、未退勤、休憩不足、未申請、未承認、長時間勤務を日次から生成 |

### フェーズ2: 月締・届出をHARMOSに近づける

| 改善 | 目的 | 画面案 | データ/処理案 |
| --- | --- | --- | --- |
| 月締状況レポート | 月締前に誰の何が残っているかを一覧する | 社員別に未申請、承認待ち、承認済み、月締状況を表示 | 日次ステータスを月単位で集計。月締不可理由も出す |
| 本人の月次確認/申請 | 職員が自分の勤怠を確認した事実を残す | 職員ポータルに `今月の勤怠を申請` | `month_close_requests` を追加。園運用で不要なら管理者締めに固定 |
| 届出モデル拡張 | 有休だけでなく勤務区分変更や時間休暇を扱う | `申請登録` に申請区分、対象日/期間、理由、現在の勤務区分を表示 | `work_procedure_requests` を追加し、承認後に日次勤怠へ反映 |
| 時間休暇対応 | 時間有給、子の看護、介護の時間単位を扱う | 申請区分で時間入力を出し分ける | 残数を日数と時間で管理。端数処理ルールは固定から開始 |

### フェーズ3: 管理者向けの精度を上げる

| 改善 | 目的 | 画面案 | データ/処理案 |
| --- | --- | --- | --- |
| 勤務区分の集計属性 | HARMOSとの差が出やすい法定集計を整える | 勤務区分詳細に所定時間、休憩、深夜、休日/休暇、集計対象を表示 | 既存勤務区分に集計用カラムを追加。まず園で使う属性だけ |
| 日次項目設定 | 権限別表示とCSV出力を調整する | 日次項目一覧に一般/承認者/管理者/CSVを持たせる | 固定項目マスタを作り、一覧とCSV生成で参照 |
| 勤怠アラート設定 | 長時間勤務や有休取得不足を設定で見られるようにする | 時間アラート/日数アラートの設定画面 | しきい値を保存し、日次・月次集計後にアラート生成 |
| CSV/PDF処理履歴 | CSV登録/出力の失敗を追跡できるようにする | `CSV・PDF履歴` 画面 | 非同期ジョブ化して、実行者、機能、成功/失敗、ファイル期限を保存 |

### フェーズ4: 後回しでよいもの

| 改善 | 理由 |
| --- | --- |
| LINE/Slack/QR/スマホ打刻 | 現在はWindows ICカード運用が中心なので優先度低 |
| 位置情報打刻 | 外出/在宅打刻を扱わないなら不要 |
| 夜勤打刻 | 園運用で日またぎがなければ不要。翌日フラグだけ将来余地として残す |
| 多段承認 | HARMOSにはあるが、1段階承認で運用できるなら固定でよい |
| オリジナル時間帯集計 | 特殊手当や独自時間帯集計が必要になるまで後回し |
| 36協定の完全再現 | 残業が少ない運用なら、まず長時間勤務アラートで十分 |

### 最初に作るならこの5つ

1. `日次勤怠修正フォーム`
2. `日次履歴タブ`
3. `勤怠エラーレポート`
4. `月締状況レポート`
5. `届出モデル拡張`

この順番なら、現場の「打刻を間違えた」「承認前に何が残っているかわからない」「月末に誰が未対応かわからない」という困りごとから先に潰せます。その後に設定系を広げると、機能が増えても運用が崩れにくいです。
