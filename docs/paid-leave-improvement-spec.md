# 有給休暇改善 詳細設計案

作成日: 2026-03-23

## 1. 目的

本書は、当園向けシステムにおける `有給休暇管理の改善仕様` を整理する。  
現行の HARMOS勤怠 運用で感じている不便さを踏まえ、以下を実現することを目的とする。

- `残数` だけではなく `付与・消化・調整` の履歴を一貫して追える
- `過去日付` を含む有給調整に対応できる
- `承認結果` と `残数反映` の関係が明確である
- 管理者が `なぜ今この残数なのか` を説明できる

## 2. 背景

HARMOS勤怠の公式FAQでは、次の点が確認できる。

- `有給、代休、特別休暇等を手動で残数調整いただける`
- `1時間単位での有給` は有料オプション
- `過去の有給残数の自動付与はできない`

このため、当園向けシステムでは `残数調整中心` の運用ではなく、`履歴中心` の設計に置き換える。

## 3. 改善方針

### 3.1 基本方針

- `現在残数` を直接編集する設計にしない
- 残数は `付与履歴 + 消化履歴 + 調整履歴` の合算で求める
- `申請中` では残数を減らさず、`承認済み` で初めて消化に反映する
- `差戻し / 却下 / 取消` 時は残数へ影響させない
- `過去日付` の付与・調整登録を可能にする
- 将来的な `時間単位有給` に拡張可能な設計にする

### 3.2 今回の対象

- 年次有給休暇
- 欠勤
- 特別休暇

### 3.3 今回の対象外

- 代休
- 振休
- フレックスタイム
- 勤務実績からの自動複雑計算

## 4. 用語定義

### 4.1 付与

法定付与や園独自付与により、有給残数を増やす処理。

### 4.2 消化

承認済みの有給申請により、有給残数を減らす処理。

### 4.3 調整

手修正や移行補正により、残数を増減する処理。

### 4.4 残数

ある基準日時における `付与 - 消化 + 調整` の結果。

## 5. 業務ルール

### 5.1 休暇区分

| 区分 | 残数管理 | 承認要否 | 備考 |
| --- | --- | --- | --- |
| 有給 | 対象 | 要 | 半日対応 |
| 欠勤 | 対象外 | 要 | 残数なし |
| 特休 | 原則対象外 | 要 | 必要なら別枠管理へ拡張 |

### 5.2 申請単位

- `1日`
- `半日(午前 / 午後)`

### 5.3 残数反映タイミング

| 状態 | 残数への影響 |
| --- | --- |
| 申請中 | なし |
| 承認 | 消化として反映 |
| 差戻し | なし |
| 却下 | なし |
| 取消 | 反映済みなら巻き戻し |

### 5.4 過去日付の扱い

- 過去の付与登録を可能にする
- 過去の調整登録を可能にする
- 過去日付の有給申請も登録可能にする
- ただし `監査ログ` と `操作理由` を必須にする

### 5.5 半日申請

- 半日申請は `開始日 = 終了日` に限定
- 消化量は `0.5日`
- `午前 / 午後` を明示する

### 5.6 将来拡張

- `時間単位` を追加する場合は `quantity_hours` を別で持つ
- 今回は `quantity_days` のみ採用する

## 6. 残数計算仕様

### 6.1 計算式

```text
残数 = 付与合計 - 消化合計 + 調整合計
```

### 6.2 付与合計

- `paid_leave_grants` の有効レコード合計

### 6.3 消化合計

- `leave_requests` のうち
  - `leave_type = PAID`
  - `status = APPROVED`
  の `quantity_days` 合計

### 6.4 調整合計

- `paid_leave_adjustments` の `adjustment_days` 合計
- 正数なら増加
- 負数なら減少

### 6.5 失効

初期版では `expires_on` を保持するが、失効自動処理は管理者運用とする。  
将来的にはバッチ処理で `失効調整` を自動登録できるようにする。

## 7. DB設計追加案

既存設計に加えて、次のテーブル追加を推奨する。

### 7.1 paid_leave_adjustments

有給残数の手動補正履歴。

| カラム名 | 型 | 説明 |
| --- | --- | --- |
| id | bigint | 主キー |
| employee_id | bigint | 職員ID |
| target_date | date | 反映基準日 |
| adjustment_days | numeric(6,2) | 増減値 |
| reason_code | varchar(30) | 調整理由区分 |
| reason_text | varchar(255) | 補足理由 |
| created_by_employee_id | bigint nullable | 職員管理者ID |
| created_by_user_id | bigint nullable | 管理者ユーザーID |
| created_at | timestamp | 登録日時 |
| updated_at | timestamp | 更新日時 |

### 7.2 paid_leave_ledger

画面表示や監査向けの有給台帳ビュー。  
物理テーブルではなく `view` またはアプリ側集約でもよい。

| 項目 | 説明 |
| --- | --- |
| employee_id | 職員ID |
| ledger_type | GRANT / CONSUME / ADJUST |
| target_date | 発生日 |
| quantity_days | 増減量 |
| source_type | REQUEST / GRANT / ADJUSTMENT |
| source_id | 元データID |
| note | 表示用補足 |

### 7.3 leave_balance_snapshots

必須ではないが、集計高速化が必要になった時点で導入する。

| 項目 | 説明 |
| --- | --- |
| employee_id | 職員ID |
| calculated_at | 算出日時 |
| balance_days | 残数 |

## 9. API追加案

### 9.1 職員向け

#### `GET /api/leave/balance`

用途:
- 現在残数と付与一覧を返す

レスポンス例:

```json
{
  "data": {
    "employeeId": 1,
    "currentBalance": 9.5,
    "grants": [
      {
        "id": 1,
        "grantedOn": "2026-01-01",
        "grantedDays": 10,
        "usedDays": 0.5,
        "expiresOn": "2027-12-31",
        "note": "初期付与"
      }
    ]
  }
}
```

#### `GET /api/leave/ledger`

用途:
- 自分の有給台帳を一覧で返す

#### `POST /api/leave/requests`

用途:
- 有給/欠勤/特休申請を登録する

補足:
- 有給のみ残数チェックを行う

#### `POST /api/leave/requests/{id}/cancel`

用途:
- 申請取消

補足:
- 承認済み取消時は消化反映を巻き戻す

### 9.2 管理者向け

#### `GET /api/admin/leave/requests`

用途:
- 申請一覧の検索・絞込

絞込条件:
- 状態
- 区分
- 期間
- 職員番号
- 職員名

#### `POST /api/admin/leave/requests/{id}/approve`

用途:
- 承認

処理:
- 状態を APPROVED に更新
- 有給なら残数へ消化反映
- 監査ログ記録

#### `POST /api/admin/leave/requests/{id}/return`

用途:
- 差戻し

#### `POST /api/admin/leave/requests/{id}/reject`

用途:
- 却下

#### `GET /api/admin/leave/ledger`

用途:
- 管理者向け有給台帳照会

#### `POST /api/admin/leave/grants`

用途:
- 有給付与登録

リクエスト例:

```json
{
  "employeeId": 1,
  "grantedOn": "2026-04-01",
  "grantedDays": 10,
  "grantType": "MANUAL",
  "expiresOn": "2028-03-31",
  "note": "年度付与"
}
```

#### `POST /api/admin/leave/adjustments`

用途:
- 有給残数調整登録

リクエスト例:

```json
{
  "employeeId": 1,
  "targetDate": "2026-03-31",
  "adjustmentDays": -0.5,
  "reasonCode": "MIGRATION_FIX",
  "reasonText": "旧システムとの差分補正"
}
```

## 11. 監査仕様

有給関連で必ず残すログ:

- 付与登録
- 調整登録
- 申請
- 承認
- 差戻し
- 却下
- 取消
- 残数巻き戻し

監査ログ項目:

- 実行者
- 実行日時
- 対象職員
- 実行種別
- 変更前値
- 変更後値
- コメント

## 12. 移行方針

### 12.1 初期移行

- 旧システムの `現在残数` をそのまま直接入れない
- `移行基準日時点の付与残数` として `MIGRATION` 付与を作る
- 必要なら差分を `MIGRATION_FIX` 調整で補正する

### 12.2 移行後

- 以後は `付与` と `調整` でのみ残数を増減する
- 休暇申請承認は `消化` として自動反映する

## 13. 実装優先順位

1. `paid_leave_adjustments` の追加
2. `有給台帳API` の追加
3. `承認時反映 / 取消時巻き戻し` の厳密化
4. `管理画面の付与・調整画面`
5. `職員向け有給台帳画面`

## 14. まず決めるべき項目

実装前に以下を確定するとブレが減る。

- 有給の付与日ルール
- 失効日ルール
- 半日区分の運用
- 過去申請の許容範囲
- 取消可能期間
- 特休を残数管理対象にするかどうか

## 15. 推奨結論

当園向けには次の形を推奨する。

- 有給は `履歴ベース`
- 欠勤は `残数なし`
- 特休は `申請管理中心`
- 半日標準対応
- 時間単位は将来拡張
- 過去調整可能
- 監査ログ必須

これにより、HARMOSよりも `説明しやすい有給運用` にできる。
